# 事件循环

## 定义

- 事件循环（Event Loop）本质上是一套任务调度机制，因为JavaScript是单线程，遇到异步操作时，必须等这些操作完成所以浏览器设计了一套分工方式：
  - `JavaScript 主线程`：负责执行同步代码
  - `浏览器/运行环境`：负责处理定时器、网络请求、DOM 事件等异步任务
  - `事件循环`：负责把“已经准备好的异步任务”放回主线程执行 
- JavaScript 是单线程的，为了避免耗时操作阻塞主线程，浏览器会把定时器、网络请求、用户事件等异步任务交给宿主环境处理。同步代码先进入调用栈执行，异步任务完成后会进入任务队列。事件循环会在调用栈清空后，从队列中取出可执行任务放回主线程。

  任务又分为宏任务和微任务。每次执行完一个宏任务后，会先清空当前产生的所有微任务，然后再执行下一个宏任务。script标签的代码就是一个宏任务，先取一个宏任务执行，如果遇到其他宏任务放到下一次事件循环执行

## 核心组成

- **调用栈（Call Stack）**：存放当前正在执行的同步代码（函数调用）。
- **宏任务队列（MacroTask Queue）**：存放 `setTimeout`、`setInterval`、I/O、UI 渲染、事件回调等。
- **微任务队列（MicroTask Queue）**：存放 `Promise.then/catch/finally`、`MutationObserver`、`queueMicrotask` 等。
- **事件循环**：不断从宏任务队列取出一个任务执行，执行完后清空微任务队列。

## 拓展

##### 调用栈与事件循环的关系

```JavaScript
console.log("1");

setTimeout(() => {
  console.log("2");
}, 1000);

console.log("3");
```

执行过程：

1. `console.log("1")` 进入调用栈，输出 `1`
2. 遇到 `setTimeout`
3. `setTimeout` 不会马上执行回调，而是把计时工作交给浏览器
4. 主线程继续执行 `console.log("3")`，输出 `3`
5. 1 秒后，浏览器把这个回调放进任务队列
6. 调用栈空了，事件循环把这个回调取出来执行，输出 `2`

## 问题延伸

#### **Q1：为什么 `setTimeout` 不精确？**

因为 `setTimeout(fn, 100)` 的意思是 **至少等待 100ms** 后把 fn 放入宏任务队列，而不是精确 100ms 后执行。如果主线程被同步代码或微任务卡住，实际执行时间会延迟。

#### **Q4：Node.js 中的事件循环和浏览器有什么区别？**

- Node.js 的宏任务分多个阶段（timers, I/O, idle, poll, check, close），并有 `process.nextTick`（优先级高于微任务）。
- 浏览器的实现更简单，只有宏任务和微任务。

#### **Q5：如何让一段代码在 DOM 更新后执行但又不被 `setTimeout` 延迟太多？**

DOM更新是同步的，修改样式但暂时不渲染，只有等到浏览器渲染时，才能看到，微任务会优先于浏览器渲染，宏任务则在浏览器渲染后。

setTimeout作为宏任务，在最后才执行，如果微任务的数量大了，延迟就会很长

答： 可以使用 Vue/React 的 `nextTick`完成DOM执行

#### **Q6：Web Worker 中的事件循环是怎样的？**

**Web Worker** 是浏览器提供的一种**多线程**机制，让你可以在**后台独立运行 JavaScript 代码，不阻塞主线程（UI 线程）**

平时你的 JS 代码跑在**主线程**上，如果主线程被一个耗时的计算卡住，Web Worker 可以开一个**子线程**，专门去跑那些耗时的计算任务，这样主线程可以继续响应用户操作，计算完成后子线程再把结果传回主线程。注意Web Worker不能操作DOM

答： Web Worker 没有浏览器渲染，没有 `window`，但它也有自己的事件循环，支持 `setTimeout`、`Promise`、`fetch` 等。
