## 代码质量检查

理解一下部分文件的作用

1. **Prettier**：统一格式化代码（自动加括号、换行、分号），让代码看起来像同一个人写的。
2. **ESLint**：检查代码中的潜在错误（比如定义了未使用的变量、`console.log` 未删除）。
3. **Husky + lint-staged**：在 `git hooks` 提交代码的一瞬间，自动运行上述检查。**如果不合格，直接阻止提交**，强制你改完再提交。
4. `eslint-plugin-vue`：让 ESLint 能看懂 Vue 文件。
5. `@typescript-eslint/*`：让 ESLint 能看懂 TypeScript。
6. `husky`：让你可以监听 Git 钩子（比如 `commit` 前）。
7. `lint-staged`：只检查你即将提交的文件（不让它跑遍整个项目，速度飞快）

husky和lint-staged可能比较陌生，用代码讲解一下

```JSON
// package.json配置文件
{
  "husky": {
    "hooks": {
      "pre-commit": "lint-staged"
      // 再commit之前触发lint-staged
    }
  },
  "lint-staged": {
    "*.js": ["eslint --fix", "git add"]
    // lint-staged对本次commit的.js文件进行eslint修复成功才自动git add，如果没有修复好就不执行git add
    //补充：prettier --write：格式化代码（缩进、引号、换行等）
    //eslint --fix：修复 ESLint 规则问题（补分号、去除未使用变量等）
  }
}
```

总的来说：**husky 就是“让 git 钩子变得好用的工具”。它让你在提交代码前自动跑检查、格式化、测试等任务，防止垃圾代码混进仓库，而且整个团队共享同一套配置。**

**问：Husky 是如何在 Git 提交前自动执行脚本的？**\
**答**：Husky 的本质是修改了项目根目录下的 `.git/hooks` 文件夹。当你运行 `husky install` 时，Husky 会在这个文件夹里注册钩子脚本。添加的 `pre-commit` 钩子，实际上就是在 `.git/hooks/pre-commit` 里写入了一条命令 `npx lint-staged`。当开发者执行 `git commit` 时，Git 会优先执行这个脚本，只有脚本退出码为 0（即检查通过）时，提交才会真正生效。

\
**问：ESLint 和 Prettier 经常互相冲突（比如缩进规则），怎么解决？**\
**答**：使用 `eslint-config-prettier` 这个插件。把它放在 `extends` 数组的**最后一位**，它会关掉所有与 Prettier 冲突的 ESLint 规则。这样 ESLint 只负责逻辑错误检查（比如变量未定义），Prettier 只负责格式（比如缩进、引号）

```JavaScript
// .eslintrc.js
module.exports = {
  extends: [
    'eslint:recommended', // 别人：检查逻辑错误
    'airbnb',             
    'prettier'            // 你：关掉所有和 Prettier 打架的 ESLint 规则，命令最后执行
  ]
};
```

## 路由懒加载

一个 Vue 项目有多个页面（比如 `Home`、`Profile`、`Order`）。如果不用懒加载，用户访问首页时，会把所有页面的 JS 一次性全下载下来，导致首屏加载变慢。

```JavaScript
// 之前（同步加载，打包进一个文件）
// import Home from '@/views/Home.vue';
// import Profile from '@/views/Profile.vue';

// 之后（异步加载，按需加载）
const routes = [
  {
    path: '/',
    component: () => import('@/views/Home.vue'), // 只有访问 / 时才加载
  },
  {
    path: '/profile',
    component: () => import('@/views/Profile.vue'), // 访问个人中心时才加载
  },
  // ...
];
```

**效果**：重新 `npm run build` 后，你会发现 `dist/assets` 里多了好几个 `.js` 文件（不再是单一的 `index.js`）。这就是被拆开的页面代码。

**问：路由懒加载（**`() => import()`**）底层原理是什么？它和常规 `import`** 有什么区别？\
**答**：常规 `import` 是静态导入，在构建时（`npm run build`）就会被分析并打包进主文件。而 `() => import()` 是动态导入，Vite 会单独为这个模块生成一个独立的 `chunk` 文件。当用户点击路由切换时，Vue Router 才会从部署了该项目的服务器去请求这个 `.js` 文件。**本质上是利用了 `Fetch`** 来异步加载远程脚本文件。

**问：除了路由懒加载，Vue 组件级别如何实现懒加载？**\
**答**：使用 Vue 提供的 `defineAsyncComponent` 函数。当某个组件在页面中不常出现（如弹窗或##复杂图表）时，用它包裹后再注册，这样组件本身的代码会在需要时才加载。

这里可能没用过会比较陌生，举个例子：

假设你有一个很大的图表组件 `HeavyChart.vue`，你不想在页面打开时就加载它，而是**等用户点击“显示图表”按钮时再加载**。

```JavaScript
<template>
  <div>
    <button @click="showChart = !showChart">切换图表</button>
    <HeavyChart v-if="showChart" />
  </div>
</template>

<script setup>
import { ref, defineAsyncComponent } from 'vue';

// 1. 用 defineAsyncComponent 包裹动态导入
const HeavyChart = defineAsyncComponent(() =>
  import('@/components/HeavyChart.vue')
);

const showChart = ref(false);
</script>
```

## 了解一下第三方库拆分（`manualChunks` 配置）

默认情况下，Vite 会把 `node_modules` 里的所有依赖都打到一个包里。如果我们把 `vue`、`element-plus`、`lodash` 拆成独立的文件，它们就能被浏览器长期缓存（因为只要版本不升级，这些文件内容就不变），**再次打包**，你会发现 `dist/assets` 里出现了 `vendor-vue.xxx.js`、`vendor-ui.xxx.js` 等。这会让浏览器缓存利用效率最大化。

**问：你提到 `manualChunks`** 拆分第三方库，这样做的利弊是什么？\
**答**：

- **好处**：利用浏览器缓存。因为 `vendor-vue` 只要你不升级 Vue 版本，文件哈希不变，用户第二次访问直接用缓存，无需重新下载。
- **坏处**：如果拆得太细（比如把几十个库全拆开），会导致浏览器发起过多 HTTP 并发请求（`http/1.1` 有并发限制），反而拖慢加载速度。因此通常只拆出高频大头（如 Vue、UI 库），其余合并到一个包中。

关于浏览器缓存：浏览器会把加载过的 JS、CSS、图片等资源**缓存到本地硬盘**。下次你访问同一个网站时，如果文件没变，浏览器就直接从硬盘读取，不再发起网络请求。

## Nginx配置防止页面404

问：如何解决线上环境404的问题呢？

**线上 SPA 页面 404 是因为 history 模式下浏览器把 `/home`** 当成真实文件路径去请求服务器，但 **`dist`** 里只有一个 **`index.html`**。解决方案是让 Nginx 把所有找不到的路径都返回 **`index.html`**，由 Vue Router 在浏览器端接管路由

在线上环境，我们通常用 Nginx 来托管（这里注意是托管，nginx不是一台服务器） `dist` 文件夹。这里面有两个雷区：

1. **History 模式 404**：Vue Router 用的是 `history` 模式，但服务器默认只会去找物理文件（`/about` 文件夹），找不到就 404。
2. **跨域代理失效**：你在 Vite 里配的 `proxy` 只在**开发环境**有效。线上必须靠 Nginx 转发请求。

这里需要扫盲一下

什么是History 和 hash模式

- Hash 模式（vueRouter默认）：`http://localhost:5173/#/home`\
  浏览器看到 `#`，知道“后面的是前端路由，不发给服务器”，所以不会 404。
- History 模式：`http://localhost:5173/home`\
  浏览器看到这是一个普通的路径 `/home`，**就会向服务器发起请求**，问“请给我 `/home` 这个文件”

什么是SPA（单页应用）

打包到服务器的dist，只有一个index.html文件，无论你访问 `/home`、`/about，`其实都应该是同一个 `index.html` 文件，由 **Vue Router 在浏览器端决定显示哪个组件**。

nginx的配置

```Nginx
server {
  listen 80;
  server_name your-domain.com;
  root /var/www/project/dist; # 指向你的打包产物（nginx会从这里找文件）

  # 处理前端路由（核心记忆点：try_files）
  location / {
    try_files $uri $uri/ /index.html; 
    # 流程：$uri 找不到 → 找 $uri/ 文件夹 → 还找不到 → 直接返回 index.html
    # $uri指的请求路径，如/home
  }

  # 接口转发（替代 Vite 的 proxy）
  location /api/ {
    proxy_pass http://your-backend-server:3000/api/; # 把 /api 转发给后端
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  # 静态资源强缓存（针对带 hash 的 js/css）
  location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

留意一下try\_files的配置：

1. 用户访问 `/home`。
2. 服务器先找 `/home` 这个文件，找不到。
3. 再找 `/home/` 这个目录，也找不到。
4. 最后返回 `/index.html`（让 Vue Router 去处理）。

<br />

问：为什么要用 `try_files` 而不是直接 rewrite？\
**你的回答**：`try_files` 效率更高。它会先尝试在磁盘上找物理文件（比如图片或字体），找不到才交给 `index.html` 处理。直接 rewrite 会丢失对静态资源的精准判断。

这里补充一下什么是rewrite

```
# 示例：把所有请求都重写为 /index.html
location / {
    rewrite ^ /index.html break;
}
```

- 用户访问 `/home` → Nginx 直接返回 `/index.html` 的内容。
- 用户访问 `/logo.png` → Nginx 依然返回 `/index.html` 的内容。

## 浏览器缓存

浏览器缓存，简单说就是“把下载过的资源存到本地，下次直接用”，**针对的是**线上生产环境\*\*。它解决的是“每次访问都要重新下载所有文件”的问题，可以显著提升页面加载速度。但缓存不能乱用，否则会导致“用户看到旧页面”的问题。浏览器缓存分为两种：**强缓存** 和 **协商缓存**。

理解何为强缓存和协商缓存

强缓存：

**规则：** 如果缓存没过期，浏览器直接从本地读取，**完全不向服务器发请求**。

**控制方式：** 服务器响应头 `Cache-Control: max-age=31536000`（单位秒）这里是一年

**效果：** 用户在 1 年内访问这个文件，浏览器直接读本地缓存，不发请求，速度极快。

协商缓存：

**规则：** 浏览器会向服务器发一个请求，询问“这个文件变了吗？如果没变，就告诉我继续用缓存”。

**控制方式：** 服务器响应头 `ETag`（文件指纹）或 `Last-Modified`（修改时间）。

**效果：** 如果文件没变，服务器返回 `304 Not Modified`，浏览器继续用缓存；如果变了，返回新文件。

他们之间的关系：**强缓存过期后走协商缓存，如果文件没变（304），服务器会同时返回新的 `Cache-Control`** 头，让浏览器重新开始强缓存。

**为什么带哈希的文件要强缓存？**

因为 Vite（或 Webpack）在打包时，会给文件加上一个**哈希值（hash）**，作为文件名的一部分，**哈希是“内容指纹”**，只要文件内容发生变化，哈希值就会变，文件名也会变。**只有 JS 和 CSS 文件有哈希值，HTML 文件没有哈希值。html作为入口需要更新（指的是**浏览器从服务器重新下载 index.html 文件），必须拿到最新版本的 **`index.html`**，才能知道去加载哪些新版本的 JS 和 CSS 文件。所以一般配置为短缓存或者不缓存

浏览器缓存的完整流程

在项目上线后，`index.html` 必须不缓存，以保证用户每次都能拿到最新版本的入口。静态资源带哈希强缓存，能充分利用浏览器缓存机制。当代码更新时，打包工具会生成新的哈希文件名，并自动更新 **`index.html`** 中的引用链接。用户刷新页面时，新的 **`index.html`** 会引导浏览器去请求这些全新的、带哈希的静态资源 URL，从而自然绕开了旧资源的强缓存，实现了无缝更新。

**问**：既然 JS 文件缓存了一年，那怎么让用户感知到代码更新了？\
**你的回答**：如果 `index.html` 不缓存，用户每次访问都会请求最新的 `index.html`，而最新的 `index.html` 里引用的就是最新带新哈希的 `app.[newhash].js`。这样就实现了“用户无感知更新”。

## **CI/CD 持续集成**

部署到服务器的项目如果项目代码发生变更，重新打包上传服务器是一件非常麻烦的事，通过githubAction配置完成持续集成（CI）和持久部署（CD）可以解决这个问题，展示一下GithubAction中的.yaml文件配置

```YAML
name: Deploy Vue App          # 这个管家的名字叫“部署Vue应用”

on:                           # 触发条件
  push:                       # 当你做出“推”这个动作
    branches: [ main ]        # 并且推的分支是“main”时

jobs: # 触发工作链路
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: 1. 拉取代码
        uses: actions/checkout@v3
      
      - name: 2. 安装依赖
        run: npm install
      
      - name: 3. 打包项目
        run: npm run build
      
        # 上面是CI下面是CD

      - name: 4. 上传到服务器（通过 SSH）
        uses: easingthemes/ssh-deploy@main
        env:
          SSH_PRIVATE_KEY: ${{ secrets.SERVER_SSH_KEY }}
          SOURCE: "dist/"
          TARGET: "/var/www/project/"
```

**问**：部署过程中，`npm install` 依赖没安装成功，流水线会怎样？\
**你的回答**：CI 流程会直接中断（报红），并且**不会**执行上传操作。这保证了线上永远不会跑一个“残缺”的代码包。
