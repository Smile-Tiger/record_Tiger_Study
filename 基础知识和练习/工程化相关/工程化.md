# 工程化（从问题中学习）

**问：路径别名 **`@`** 的原理是什么？它在生产环境（打包后）还能用吗？**\
**答**：`@` 是构建工具（Vite/Webpack）在**编译阶段**做的“字符串替换”。当 Vite 解析代码时，看到 `@/`，就会在内存中把它映射成 `path.resolve(__dirname, 'src')` 这个绝对路径。**它能用**，因为在执行 `npm run build` 打包时，Vite 会把这个别名映射关系固化到最终的产物中，所以线上不会报错。

 

**问：环境变量为什么在 **`npm run build`** 后就被“写死”了？如果我想在运行时动态改地址怎么办？**\
**答**：因为环境变量是在**构建阶段（Build Time）**被注入的。Vite 会做“静态替换”，把代码里的 `import.meta.env.VITE_API_URL` 直接替换成具体的字符串（比如 `"https://xxx"`）。**如果想在运行时（Runtime）动态改**，就不能用 `.env`，而要用一个 `config.js` 文件在 `public` 目录下，或者在 `window` 全局对象上挂载变量，或者请求后端的配置接口。

问题词汇扫盲

构建阶段：**就是你把代码“打包”成最终部署文件的那个时刻。**你写的源代码是**一堆零配件，**你执行 `npm run build`，Vite 开始工作，把零配件**组装成一个完整的机器**（`dist` 文件夹里的静态文件：`index.html`、`app.js`、`app.css` 等）**，在这个阶段，Vite 会做“静态替换”：**

```JavaScript
// 你在代码里写的是：
const url = import.meta.env.VITE_API_URL;

// npm run build 之后，Vite 把它变成了：
const url = "https://api.example.com";
```

运行时：**用户已经拿到打包好的文件，在浏览器里打开、运行的时刻。如果 **`app.js`** 里已经写死了 **`"https://api.example.com"`**，用户访问时就只能连这个地址，无法动态改变。**

环境变量：根据不同的命令去加载不同的环境变量，完成不同的替换

- `.env.development`（开发环境，运行 `npm run dev` 时加载）
- `.env.production`（生产环境，打包 `npm run build` 时加载）

Vite 规定，只有以 `VITE_` 开头的变量才会暴露给前端代码。

npm run dev和npm run build的区别

| 命令 | 阶段 | 替换方式 | 结果 |
|------|------|----------|------|
| `npm run dev` | 开发服务器（Dev Server） | **运行时注入** | Vite 启动一个开发服务器，在内存中把 `import.meta.env.VITE_API_URL` 替换成对应的值，但**不会改动硬盘上的源文件**。 |
| `npm run build` | 构建打包（Build） | **静态替换** | Vite 把代码里的 `import.meta.env.VITE_API_URL` **物理替换**成具体字符串，写入最终的 `dist/app.js` 文件里。 |

<br />

**问：Vite 代理为什么能解决跨域，而前端直接用 **`fetch`** 不行？**\
**答**：跨域（CORS）是**浏览器的安全策略**限制的。当你在浏览器里直接请求不同域名的接口，浏览器会拦截。而 Vite 代理是**服务器转发**：浏览器请求的是同源的 Vite 开发服务器（`localhost:5173`），不触发跨域限制。Vite 开发服务器再去请求后端（`localhost:3000`）是服务器之间的通信，**不受同源策略约束**。拿到数据后，Vite 再原样吐给浏览器。
