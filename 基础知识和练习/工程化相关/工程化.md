# 工程化（从问题中学习）

**问：路径别名 `@`** 的原理是什么？它在生产环境（打包后）还能用吗？\
**答**：`@` 是构建工具（Vite/Webpack）在**编译阶段**做的“字符串替换”。当 Vite 解析代码时，看到 `@/`，就会在内存中把它映射成 `path.resolve(__dirname, 'src')` 这个绝对路径。**它能用**，因为在执行 `npm run build` 打包时，Vite 会把这个别名映射关系固化到最终的产物中，所以线上不会报错。

 

**问：环境变量为什么在 `npm run build`** 后就被“写死”了？如果我想在运行时动态改地址怎么办？\
**答**：因为环境变量是在\*\*构建阶段（Build Time）\*\*被注入的。Vite 会做“静态替换”，把代码里的 `import.meta.env.VITE_API_URL` 直接替换成具体的字符串（比如 `"https://xxx"`）。**如果想在运行时（Runtime）动态改**，就不能用 `.env`，而要用一个 `config.js` 文件在 `public` 目录下，或者在 `window` 全局对象上挂载变量，或者请求后端的配置接口。

问题词汇扫盲

构建阶段：**就是你把代码“打包”成最终部署文件的那个时刻。你写的源代码是**一堆零配件，**你执行 `npm run build`，Vite 开始工作，把零配件**组装成一个完整的机器（`dist` 文件夹里的静态文件：`index.html`、`app.js`、`app.css` 等）**，在这个阶段，Vite 会做“静态替换”：**

```JavaScript
// 你在代码里写的是：
const url = import.meta.env.VITE_API_URL;

// npm run build 之后，Vite 把它变成了：
const url = "https://api.example.com";
```

运行时：**用户已经拿到打包好的文件，在浏览器里打开、运行的时刻。如果 `app.js`** 里已经写死了 **`"https://api.example.com"`**，用户访问时就只能连这个地址，无法动态改变。

环境变量：根据不同的命令去加载不同的环境变量，完成不同的替换

- `.env.development`（开发环境，运行 `npm run dev` 时加载）
- `.env.production`（生产环境，打包 `npm run build` 时加载）

Vite 规定，只有以 `VITE_` 开头的变量才会暴露给前端代码。

npm run dev和npm run build的区别

| 命令            | 阶段                     | 替换方式       | 结果                                                                                                              |
| --------------- | ------------------------ | -------------- | ----------------------------------------------------------------------------------------------------------------- |
| `npm run dev`   | 开发服务器（Dev Server） | **运行时注入** | Vite 启动一个开发服务器，在内存中把 `import.meta.env.VITE_API_URL` 替换成对应的值，但**不会改动硬盘上的源文件**。 |
| `npm run build` | 构建打包（Build）        | **静态替换**   | Vite 把代码里的 `import.meta.env.VITE_API_URL` **物理替换**成具体字符串，写入最终的 `dist/app.js` 文件里。        |

**问：Vite 代理为什么能解决跨域，而前端直接用 `fetch`** 不行？\
**答**：跨域（CORS）是**浏览器的安全策略**限制的。当你在浏览器里直接请求不同域名的接口，浏览器会拦截。而 Vite 代理是**服务器转发**：浏览器请求的是同源的 Vite 开发服务器（`localhost:5173`），不触发跨域限制。Vite 开发服务器再去请求后端（`localhost:3000`）是服务器之间的通信，**不受同源策略约束**。拿到数据后，Vite 再原样吐给浏览器。

问题扫盲

什么是跨域？为什么会出现跨域？

跨域是因为浏览器的同源策略限制，只有当协议、域名、端口号完全相同时的两个URL才能称为同源。举个例子：

- 前端跑在 `http://localhost:5173`（Vite 默认端口）

* 后端跑在 `http://localhost:3000`（你的后端服务端口）
* 前端用 `fetch('http://localhost:3000/users')` → **跨域！**

很多人误以为“浏览器没发请求”，其实**请求是发出去了，后端也处理了，只是浏览器把响应给拦住了**，不让你拿到数据。

怎么解决跨域？

后端配置CORS：后端在响应头里告诉浏览器“这个前端可以访问我的数据”（最正规、推荐）

```JavaScript
// node.js + express
const cors = require('cors');
app.use(cors()); // 允许所有来源
// 或指定允许的来源
app.use(cors({ origin: 'http://localhost:5173' }));
```

Vite 配置 proxy 代理（常用于开发环境）

```JavaScript
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
        // 将/api 替换为空字符串，只转发/user拼接到后端地址的后面
      },
    },
  },
};
```

- 前端请求 `fetch('/api/users')` → 请求的是 `http://localhost:5173`（同源，不跨域）
- Vite 服务器看到 `/api`，就去请求到 `http://localhost:3000/users`
- 服务器之间的通信没有跨域限制
- Vite 拿到数据后，再原样返回给浏览器

> **注意：** 这个只在开发环境有效（`npm run dev`），打包后就不起作用了。

Nginx反向代理（常用于生产环境）

把前端和后端放在同一个域名下，比如：

- 前端：`https://yourapp.com`
- 后端：`https://yourapp.com/api`

通过 Nginx 配置把 `/api` 的请求转发到后端的真实地址，对浏览器来说，所有请求都是同源的，没有跨域问题。

**问：Vite 和 Webpack 开发模式最大的区别是什么？为什么 Vite 启动那么快？**\
**答**：Webpack 是 **“Bundle（打包）”模式**，启动时要先把你整个项目的所有 JS 文件打包成一个 bundle，再启动服务器，所以项目越大，启动越慢（需要等全部打包完）。Vite 是 **“ESM（原生模块）”模式**，启动时**不打包**，直接启动服务器，利用浏览器支持 `<script type="module">` 的特性，按需加载依赖（预构建），所以启动速度极快，且热更新只更新修改的文件，不会全量刷新。

问题扫盲

先Vite 和 Webpack的启动是什么

webpack

1. 你运行 `npm run dev`。
2. Webpack 会从入口文件开始，**把整个项目所有的文件（JS、CSS、图片、Vue/React 组件）全部分析、编译、打包**，生成一个完整的 `bundle.js`。
3. 打包完成之后，才启动开发服务器。
4. 你修改代码时，Webpack 会**重新打包**整个依赖图

vite

1. 你运行 `npm run dev`。
2. Vite **不打包**，它直接把开发服务器启动起来（几乎瞬间）。
3. 浏览器请求哪个文件，Vite 就**按需编译**哪个文件（利用浏览器原生 ES Module 支持）。
4. 你修改代码时，Vite 只**重新编译你改的那个文件**，并通过 HMR（热模块替换）更新浏览器。

为什么webpack慢，为什么Vite那么快？

Webpack 内部就有，它的核心工作是：

> 把所有文件（JS、CSS、图片）都当成模块，通过 `require` / `import` 关系，分析依赖图，最终**合并成少数几个文件**（`bundle.js`）。

这个过程在开发阶段也要做，因为 Webpack 的 HMR 需要基于打包后的模块。

Vite 利用了**浏览器原生 ES Module 支持**。

- 浏览器遇到 `import` 语句，会发出 HTTP 请求去加载模块。
- Vite 开发服务器只做一件事：**拦截这些请求，实时编译并返回**。
- 没有打包步骤，所以启动和热更新都极快。
